# User Story - US_025

## Story ID
* ID: US_025

## Story Title
* Admin User Lifecycle Management

## Description
* As an admin, I want to create, update, disable, and role-assign user accounts, so that I can govern platform access and ensure proper role-based privileges for all users.

## Acceptance Criteria
1. **Given** I am authenticated as an admin, **When** I access the user management panel, **Then** I see a searchable, filterable table of all user accounts with name, email, role, and status per FR-015.

2. **Given** I click "Create User", **When** I submit a valid username, role, and permissions, **Then** the system creates the account via ASP.NET Core Identity, persists the Staff or Admin record, and records the action in the audit log per UC-006.

3. **Given** I select a user and change their role, **When** I confirm the role change, **Then** the system updates role claims, modifies permissions_bitfield, invalidates the user's session cache, and the user must re-login per UC-006.

4. **Given** I click "Disable" on a user account, **When** the action is confirmed, **Then** the system sets is_active to false, disables authentication, terminates active sessions, and blocks all future login attempts per FR-015.

5. **Given** I attempt to assign conflicting role combinations, **When** the validation check runs, **Then** the system blocks the change and displays the specific conflict per UC-006 extension 2a.

## Edge Cases
* What happens when an admin tries to disable their own account? (System blocks self-disable action to prevent admin lockout)
* How does the system handle disabling a user who is currently active? (System terminates active sessions immediately; user sees "Your account has been disabled" on next action)
