# User Story - US_021

## Story ID
* ID: US_021

## Story Title
* 360-Degree Patient View Assembly

## Description
* As a staff member, I want to view a consolidated 360-degree patient summary assembled from intake data, uploaded documents, and extracted facts with de-duplication and source traceability, so that I can verify patient information efficiently before a visit.

## Acceptance Criteria
1. **Given** a patient has extracted facts from multiple documents, **When** I open the patient chart, **Then** the system assembles a consolidated 360-degree view with de-duplicated facts grouped by category (vitals, medications, history, diagnoses, procedures) per FR-012.

2. **Given** overlapping facts exist across sources, **When** the view is assembled, **Then** the system de-duplicates using semantic similarity and entity resolution, presenting each consolidated value with all contributing source references per AIR-003.

3. **Given** a fact is displayed in the 360-view, **When** I click on the fact, **Then** the system shows the source document segment with the original text highlighted at character-level precision per FR-012.

4. **Given** the PatientView360 record exists, **When** concurrent staff members attempt updates, **Then** optimistic concurrency control via version field prevents race conditions per DR-018.

5. **Given** healthcare semantic colors are applied, **When** viewing fact categories, **Then** vitals appear in pink, medications in orange, history in brown, diagnoses in purple, and procedures in teal.

## Edge Cases
* What happens when no extracted data is available? (System displays "No extracted data available" empty state and keeps the case in pending review status per UC-005 extension 1a)
* How does the system handle conflicting fact values? (Conflicts are detected and flagged for mandatory staff review per FR-013 - handled in US_022)
