# User Story - US_030

## Story ID
* ID: US_030

## Story Title
* Azure OpenAI Client & Gateway Middleware

## Description
* As a backend developer, I want a centralized Azure OpenAI client with gateway middleware that handles authentication, request routing, token budgeting, and retry logic, so that all AI features consume a single, reliable integration point.

## Acceptance Criteria
1. **Given** an AI feature needs to call Azure OpenAI, **When** it sends a request through the gateway, **Then** the gateway authenticates with Azure OpenAI using managed identity credentials and routes to the configured model endpoint (GPT-4 Turbo for inference, text-embedding-3-small for embeddings) per TR-006 and TR-007.

2. **Given** the gateway receives a request, **When** it processes the request, **Then** it enforces token budget limits (max 4096 output tokens per request) and appends system prompts appropriate to the feature context per AIR-O01.

3. **Given** an Azure OpenAI call fails with a transient error (429, 503), **When** the retry policy activates, **Then** the gateway retries up to 3 times with exponential backoff (1s, 2s, 4s) before returning an error per AIR-S03.

4. **Given** all AI requests, **When** they pass through the gateway, **Then** request/response metadata (model, tokens_used, latency_ms, status) is logged for observability without logging PHI content per TR-006.

5. **Given** the Azure OpenAI endpoint is unreachable, **When** the health check fails, **Then** the gateway marks the service as degraded and AI features gracefully degrade (return cached responses or "AI temporarily unavailable" messages) per AIR-S03.

## Edge Cases
* What happens when token budget is exceeded mid-response? (Gateway truncates response at budget limit and marks response as truncated)
* How does the system handle model version changes? (Configuration-driven model selection; version pinned in settings, updated via deployment)
