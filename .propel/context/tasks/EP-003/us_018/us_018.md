# User Story - US_018

## Story ID
* ID: US_018

## Story Title
* Document Upload & Storage

## Description
* As a patient, I want to upload historical and post-visit documents and have them associated with my patient record and encounter context, so that my clinical history is available for staff review and AI extraction.

## Acceptance Criteria
1. **Given** I have an active patient profile, **When** I navigate to the document upload screen, **Then** I see a drag-and-drop upload zone that accepts PDF files up to 25MB per FR-010.

2. **Given** I select files for upload, **When** the upload begins, **Then** per-file progress bars display upload status and the system validates file type and size before persistence.

3. **Given** a valid document is uploaded, **When** the upload completes, **Then** the original file is stored with a file reference URI, a ClinicalDocument record is created with patient and optional encounter association, and an audit log entry records the upload event.

4. **Given** an invalid file type is uploaded, **When** validation fails, **Then** the system displays "Unsupported file type" error and rejects the file without creating a document record.

5. **Given** the upload completes, **When** the document is persisted, **Then** a background extraction job is queued for processing and the document status shows "queued" in my document list.

## Edge Cases
* What happens when the file exceeds 25MB? (Validation toast: "File size must be under 25MB. Please select a smaller file." per figma_spec edge case)
* How does the system handle concurrent uploads of multiple files? (Each file processes independently with its own progress bar and status tracking)
