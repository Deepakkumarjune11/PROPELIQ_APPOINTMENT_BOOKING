# User Story - US_019

## Story ID
* ID: US_019

## Story Title
* Document Chunking & Embedding Pipeline

## Description
* As a system, I want to parse uploaded PDF documents, chunk them into 512-token segments with 25% overlap, and generate vector embeddings stored in pgvector, so that the RAG retrieval pipeline can perform similarity-based document search.

## Acceptance Criteria
1. **Given** a document extraction job is dequeued, **When** the background worker processes the document, **Then** the PDF is parsed to text and chunked into 512-token segments with 25% overlap per AIR-R01.

2. **Given** document chunks are created, **When** embedding generation runs, **Then** each chunk produces a 1536-dimensional vector using text-embedding-3-small and is stored in the pgvector table with document_id, chunk_index, chunk_text, and token_count per DR-016.

3. **Given** embeddings are stored in pgvector, **When** a similarity search query is executed, **Then** cosine distance metric returns relevant chunks within <10ms at p95 per TR-015.

4. **Given** the document is being processed, **When** the extraction_status is updated, **Then** the ClinicalDocument record transitions from "queued" to "processing" and UI polling reflects the status change.

## Edge Cases
* What happens when a PDF contains no extractable text (scanned image)? (System detects empty text output, sets extraction_status to "manual_review", and flags for staff)
* How does the system handle very large documents (100+ pages)? (Chunking proceeds normally; pgvector stores all resulting chunks with sequential chunk_index values)
