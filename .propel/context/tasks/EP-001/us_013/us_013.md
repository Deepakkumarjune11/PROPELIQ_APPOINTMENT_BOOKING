# User Story - US_013

## Story ID
* ID: US_013

## Story Title
* No-Show Risk Scoring & Booking Transaction

## Description
* As a clinic operator, I want the system to calculate a rule-based no-show risk score during booking and create the appointment atomically, so that staff can prioritize follow-up for high-risk patients and the booking is securely committed.

## Acceptance Criteria
1. **Given** a patient is completing booking, **When** the appointment transaction is initiated, **Then** the system calculates a no-show risk score using configurable scheduling signals (time-to-appointment, day-of-week) and patient-response signals (insurance status, intake completion) per FR-006.

2. **Given** the no-show risk score is calculated, **When** the score exceeds 70%, **Then** the slot selection displays an orange warning badge "High no-show risk detected" with a tooltip explaining the contributing factors.

3. **Given** all booking details are valid, **When** the booking transaction executes, **Then** the patient record, appointment, intake response, insurance validation, and no-show score are committed atomically in a single database transaction.

4. **Given** the slot was claimed by another patient between selection and booking, **When** the transaction attempts to reserve the slot, **Then** the system returns a 409 Conflict and the UI reverts to slot selection with a message "Slot no longer available."

## Edge Cases
* What happens when risk scoring signals are incomplete? (System applies available signals only and notes "partial scoring" in the risk assessment metadata)
* How does the system handle concurrent booking attempts for the same slot? (Database-level row locking on the appointment slot prevents double-booking; second transaction receives 409)
