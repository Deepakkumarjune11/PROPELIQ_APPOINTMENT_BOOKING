# User Story - US_015

## Story ID
* ID: US_015

## Story Title
* Preferred Slot Swap Watchlist

## Description
* As a patient, I want to select a preferred unavailable slot and have the system automatically swap my appointment when that slot opens, so that I can get my ideal appointment time without manual monitoring.

## Acceptance Criteria
1. **Given** I have a confirmed appointment, **When** I navigate to preferred slot selection, **Then** I see a calendar showing unavailable slots that are eligible for watchlist enrollment per FR-004.

2. **Given** I select a preferred unavailable slot, **When** I confirm the selection, **Then** the system registers the appointment on the swap watchlist with preferred_slot_id updated in the Appointment record.

3. **Given** my appointment is on the watchlist, **When** the preferred slot becomes available, **Then** the system atomically moves my appointment to the preferred slot, releases the original slot, and notifies me via SMS/email per FR-005.

4. **Given** the swap is executed, **When** viewing my appointments, **Then** the appointment shows the new slot datetime and the watchlist status is cleared.

5. **Given** another patient claims the preferred slot first, **When** the swap check runs, **Then** my original appointment remains unchanged and the watchlist entry is preserved for continued monitoring.

## Edge Cases
* What happens when the preferred slot is no longer eligible? (System keeps original appointment, removes watchlist entry, and notifies patient that their preferred slot is no longer available)
* How does the system prevent double-booking during atomic swap? (Swap executes within a database transaction: lock preferred slot → update appointment → release original slot → remove from watchlist → commit)
* What happens when notification delivery fails after swap? (Swap is committed, failure logged, notification retried via background job)
