# User Story - US_007

## Story ID
* ID: US_007

## Story Title
* Views, Codes & Audit Entity Layer

## Description
* As a developer, I want PatientView360, CodeSuggestion, and AuditLog entities implemented in EF Core, so that clinical verification, code review, and compliance audit workflows have proper data persistence.

## Acceptance Criteria
1. **Given** the PatientView360 entity is configured, **When** storing a consolidated patient view, **Then** the system persists id (UUID PK), patient_id (FK), consolidated_facts (JSONB), conflict_flags (string array), verification_status, last_updated, and version (int for optimistic concurrency) per DR-006.

2. **Given** the CodeSuggestion entity is configured, **When** storing a code suggestion, **Then** the system persists id (UUID PK), patient_id (FK), code_type, code_value, evidence_fact_ids (UUID array), staff_reviewed (boolean), review_outcome, created_at, and reviewed_at per DR-007.

3. **Given** the AuditLog entity is configured as immutable append-only, **When** attempting to update or delete an audit log record, **Then** the database rejects the operation, preserving immutability per DR-008.

4. **Given** soft delete is implemented for patient records, **When** a patient is soft-deleted, **Then** related PatientView360 records are preserved for audit trail per DR-017.

5. **Given** the AuditLog entity exists, **When** inserting an audit record, **Then** actor_id, actor_type, action_type, target_entity_type, target_entity_id, payload (JSONB), and created_at are persisted per DR-008.

## Edge Cases
* What happens with concurrent PatientView360 updates? (Optimistic concurrency using version field detects conflicts and returns 409 to the caller)
* How does the system handle large JSONB payloads in consolidated_facts? (PostgreSQL JSONB has no practical size limit but API layer enforces max payload validation)
