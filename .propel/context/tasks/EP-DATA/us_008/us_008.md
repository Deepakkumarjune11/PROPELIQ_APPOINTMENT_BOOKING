# User Story - US_008

## Story ID
* ID: US_008

## Story Title
* Staff, Admin Entities & Seed Data

## Description
* As a developer, I want Staff and Admin entities with seed data and migration scripts, so that role-based operations and development testing have baseline data available from initial deployment.

## Acceptance Criteria
1. **Given** the Staff entity is configured, **When** creating a staff record, **Then** the system persists id (UUID PK), username, role (enum: front-desk, call-center, clinical-reviewer), permissions_bitfield (int), auth_credentials, created_at, and is_active per DR-009.

2. **Given** the Admin entity is configured, **When** creating an admin record, **Then** the system persists id (UUID PK), username, access_privileges (int), auth_credentials, created_at, and is_active per DR-010.

3. **Given** referential integrity constraints are configured, **When** a Staff or Admin user creates audit entries, **Then** the AuditLog.actor_id correctly references the originating entity per DR-011.

4. **Given** seed data is configured, **When** running database migrations in development, **Then** test patients, staff users, admin users, and sample appointments are created for immediate development and testing use.

## Edge Cases
* What happens when seeding data in a production environment? (Seed data execution is conditional on environment configuration, skipped in production)
* How does the system handle permissions_bitfield overflow? (Bitfield is stored as int with defined max bits; API validates permission combinations before persistence)
