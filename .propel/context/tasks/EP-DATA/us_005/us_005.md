# User Story - US_005

## Story ID
* ID: US_005

## Story Title
* Patient & Appointment Entity Layer

## Description
* As a developer, I want Patient and Appointment entities with all attributes, relationships, and constraints implemented in EF Core, so that booking workflows have a persisted data foundation with referential integrity.

## Acceptance Criteria
1. **Given** the Patient entity is configured, **When** creating a patient record, **Then** the system persists id (UUID PK), email (unique key), name, DOB, phone, insurance_provider, insurance_member_id, insurance_status, created_at, updated_at, and is_deleted fields per DR-001.

2. **Given** the Appointment entity is configured, **When** creating an appointment, **Then** the system persists id (UUID PK), patient_id (FK), slot_datetime, status (enum: booked, arrived, completed, cancelled, no-show), preferred_slot_id (self-referencing FK), no_show_risk_score, created_at, updated_at, and is_deleted fields per DR-002.

3. **Given** referential integrity is enforced, **When** attempting to create an appointment without a valid patient_id, **Then** the database rejects the operation with a foreign key constraint violation per DR-011.

4. **Given** soft delete is implemented, **When** deleting a patient or appointment record, **Then** the is_deleted flag is set to true and the record is excluded from default queries but preserved in the database per DR-017.

5. **Given** the Patient-Appointment relationship exists, **When** querying a patient, **Then** the system can navigate to all related appointments via one-to-many relationship.

## Edge Cases
* What happens when inserting a patient with a duplicate email? (Database rejects with unique constraint violation, API returns 409 Conflict)
* How does the appointment self-reference work for preferred slots? (preferred_slot_id is nullable FK pointing to same Appointment table, allowing one preferred slot per appointment)
