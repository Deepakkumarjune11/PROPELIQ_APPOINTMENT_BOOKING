# User Story - US_006

## Story ID
* ID: US_006

## Story Title
* Clinical & Intake Entity Layer

## Description
* As a developer, I want IntakeResponse, ClinicalDocument, and ExtractedFact entities implemented in EF Core, so that clinical document processing and intake workflows have proper persistence with source traceability.

## Acceptance Criteria
1. **Given** the IntakeResponse entity is configured, **When** storing an intake submission, **Then** the system persists id (UUID PK), patient_id (FK), mode (conversational/manual), answers (JSONB), and created_at per DR-003.

2. **Given** the ClinicalDocument entity is configured, **When** uploading a document, **Then** the system persists id (UUID PK), patient_id (FK), encounter_id (optional FK), file_reference, extraction_status, uploaded_at, and processed_at per DR-004.

3. **Given** the ExtractedFact entity is configured, **When** storing an extracted fact, **Then** the system persists id (UUID PK), document_id (FK), fact_type (enum: vitals, medications, history, diagnoses, procedures), value, confidence_score (float), source_char_offset (int), source_char_length (int), and extracted_at per DR-005.

4. **Given** ClinicalDocument-ExtractedFact relationship exists, **When** querying a document, **Then** the system can navigate to all extracted facts via one-to-many relationship with proper FK constraint per DR-011.

## Edge Cases
* What happens when storing JSONB answers with deeply nested structure? (EF Core serializes JSONB with configured max depth; oversized payloads rejected at API validation layer)
* How does the system handle documents with no encounter association? (encounter_id is nullable, allowing pre-visit historical document uploads without encounter context)
