# User Story - US_035

## Story ID
* ID: US_035

## Story Title
* Performance Tuning & Resilience Patterns

## Description
* As a backend developer, I want performance tuning strategies including query optimization, connection pooling, caching patterns, and resilience mechanisms, so that the platform handles 200 concurrent users with sub-second API response times.

## Acceptance Criteria
1. **Given** the platform is under load, **When** 200 concurrent users are active, **Then** API p95 response time remains below 500ms for standard CRUD operations per NFR-011.

2. **Given** database queries execute, **When** query execution is monitored, **Then** all queries complete within 100ms (simple) or 500ms (complex joins/aggregations) through proper indexing and query optimization per NFR-016.

3. **Given** frequently accessed data (provider schedules, slot availability), **When** a read request is made, **Then** the system serves from Upstash Redis cache with TTL-based invalidation, achieving > 90% cache hit ratio per TR-020.

4. **Given** an external dependency (Azure OpenAI, email service) fails, **When** the failure is detected, **Then** the system applies bulkhead isolation (dedicated thread pools per dependency) and graceful degradation per TR-021 and TR-023.

5. **Given** the platform is load tested, **When** traffic ramps from 50 to 200 concurrent users, **Then** response times degrade linearly (not exponentially) and no requests are dropped per NFR-011.

## Edge Cases
* What happens when the Redis cache becomes unavailable? (System falls back to direct database queries with slightly higher latency; alert triggers for cache restoration)
* How does the system handle memory pressure under sustained load? (Connection pool limits, GC tuning, and memory guards prevent OOM; horizontal scaling triggers at 80% memory)
