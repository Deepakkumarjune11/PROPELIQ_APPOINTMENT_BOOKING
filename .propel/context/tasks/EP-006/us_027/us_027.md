# User Story - US_027

## Story ID
* ID: US_027

## Story Title
* PHI Encryption & Data Protection

## Description
* As a security engineer, I want all Protected Health Information (PHI) encrypted at rest with AES-256 and in transit with TLS 1.3, so that patient data is safeguarded against unauthorized disclosure and meets HIPAA encryption standards.

## Acceptance Criteria
1. **Given** PHI data (patient records, clinical notes, insurance info) is written to PostgreSQL, **When** the write operation completes, **Then** column-level encryption using AES-256-GCM is applied to designated PHI fields per NFR-003.

2. **Given** any API communication between client and server, **When** the request is transmitted, **Then** TLS 1.3 is enforced and connections using TLS < 1.3 are rejected per DR-015.

3. **Given** encryption keys exist, **When** key rotation is triggered (every 90 days or on-demand), **Then** the system re-encrypts affected data with the new key without downtime and invalidates old keys per TR-022.

4. **Given** database backups are created, **When** the backup is stored, **Then** the backup files are encrypted using the same AES-256 standard and are inaccessible without the current key per NFR-003.

5. **Given** a developer queries the database directly, **When** they read a PHI column, **Then** the raw value is ciphertext; decryption only occurs through the application layer with proper authorization.

## Edge Cases
* What happens if key rotation fails mid-process? (System rolls back to previous key; alerts security team; marks rotation as failed for retry)
* How does the system handle encryption for search operations? (Deterministic encryption or encrypted search indexes are used for lookup fields)
