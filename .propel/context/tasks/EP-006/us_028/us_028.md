# User Story - US_028

## Story ID
* ID: US_028

## Story Title
* Backup Strategy & Migration Patterns

## Description
* As a DevOps engineer, I want automated, encrypted backups with point-in-time recovery and versioned EF Core migrations, so that I can recover from data loss within defined RPO/RTO targets and safely evolve the database schema.

## Acceptance Criteria
1. **Given** the production database, **When** the backup schedule triggers (every 4 hours), **Then** an encrypted incremental backup is created and stored in geographically redundant storage per DR-013.

2. **Given** a backup exists, **When** a restore is requested, **Then** the system supports point-in-time recovery to any second within the last 7 days per DR-013.

3. **Given** a backup restore is performed, **When** the restored database comes online, **Then** all data integrity constraints, indexes, and referential integrity are validated automatically per DR-013.

4. **Given** a schema change is needed, **When** a developer creates an EF Core migration, **Then** the migration follows the version-controlled, idempotent pattern with rollback scripts and is applied via CI/CD per DR-014.

5. **Given** a migration fails during deployment, **When** the failure is detected, **Then** the system automatically rolls back to the previous migration version and alerts the DevOps team per DR-014.

## Edge Cases
* What happens when backup storage is full? (System alerts DevOps team; falls back to full backup to alternative storage; older archives are compressed)
* How does the system handle long-running migrations on large tables? (Use online schema change tools; apply non-blocking ALTER operations)
