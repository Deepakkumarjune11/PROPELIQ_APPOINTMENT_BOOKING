# User Story - US_026

## Story ID
* ID: US_026

## Story Title
* Immutable Audit Log & Compliance Logging

## Description
* As a compliance officer, I want all data access, modifications, and security events recorded in an immutable, append-only audit log with tamper-detection, so that I can demonstrate HIPAA compliance and investigate security incidents.

## Acceptance Criteria
1. **Given** any user creates, reads, updates, or deletes a record, **When** the operation completes, **Then** the system writes an entry to the audit_logs table containing user_id, action, entity_type, entity_id, old_values, new_values, ip_address, and timestamp per FR-016 and DR-012.

2. **Given** the audit_logs table, **When** any attempt to UPDATE or DELETE a row is made, **Then** the database rejects the operation (append-only constraint enforced via database triggers/policies) per NFR-007.

3. **Given** audit records exist, **When** I query the audit log with date range, user, entity type, or action filters, **Then** the system returns paginated results within 2 seconds for up to 1M records per NFR-013.

4. **Given** an audit entry is created, **When** the entry is persisted, **Then** a SHA-256 hash chain links each entry to its predecessor ensuring tamper detection per TR-018.

5. **Given** the audit subsystem experiences a write failure, **When** the main operation is in flight, **Then** the main operation is rolled back (audit write is part of the same transaction boundary).

## Edge Cases
* What happens when the audit log table grows beyond retention limits? (System archives entries older than 7 years to cold storage per DR-012)
* How does the system handle concurrent audit writes during high-load periods? (Database serialization via row-level locking ensures ordering; no entries are dropped)
